# 개발 컨벤션

개인 MVP 개발 단계 기준으로 작성된 최소 규칙입니다.

---

## 브랜치 네이밍

```
feat/이슈번호-기능명
fix/이슈번호-수정명
chore/이슈번호-작업명
```

**예시**
```
feat/12-player-attack
fix/7-item-null-pointer
chore/3-h2-datasource-config
```

---

## 커밋 메시지

```
타입: 변경 내용 요약
```

| 타입 | 사용 상황 |
|------|-----------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `chore` | 설정, 문서, 빌드 작업 |
| `refactor` | 기능 변경 없는 코드 개선 |
| `docs` | 문서 수정 |
| `test` | 테스트 추가/수정 |

**예시**
```
feat: 플레이어 공격 기능 추가
fix: 아이템 조회 시 NPE 수정
chore: H2 datasource 설정 추가
refactor: PlayerService 메서드 분리
```

---

## 작업 흐름

1. **GitHub Issue 생성** — 작업 내용을 이슈로 먼저 등록
2. **브랜치 생성** — 이슈 번호 기준으로 브랜치 생성
3. **작업 및 커밋** — 단위별로 커밋
4. **PR 생성** — PR 템플릿에 맞게 작성
5. **셀프 리뷰 후 main 병합**

```
main ← feat/12-player-attack
         ↑
     (PR + 셀프 리뷰)
```

---

## 푸시 요청 규칙

"커밋하고 올려줘", "푸시해줘" 등의 요청을 받으면, 실행 전에 아래 내용을 한 번에 확인한다.

> `add → commit → push` 진행할게요. 커밋 메시지: `[타입]: [내용]` — 진행할까요?

확인 후 한 번에 처리한다. 단계별로 나눠서 물어보지 않는다.

### 문서 작업 예외

`docs` 타입의 문서 작업(README, CONVENTION, PRODUCT 등 md 파일 수정)은 이슈를 생성하지 않고 main에 직접 푸시할 수 있다.

---


## PR 작성 기준

- **작업 내용**: 이번 PR에서 무엇을 했는지
- **테스트 방법**: 어떻게 확인했는지 (API 호출, 단위 테스트 등)
- **관련 이슈**: `closes #이슈번호`
- **남은 작업**: 이번 PR에 포함하지 않은 후속 작업
